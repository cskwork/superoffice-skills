# biz-report - 비즈니스 보고 작법 (경영진 대상 메시지 구조)

Load from `office.md` / `pptx.md` / `docx.md` / `xlsx.md` when 업무 보고 문서·덱을 만들 때. 디자인(색/레이아웃)은 `pptx.md`, 여기는 **메시지 구조**다 - 둘 다 적용해야 비즈니스 문서가 된다. 예쁜 색만으로는 부족하고, 결론이 먼저 보여야 한다.

## 핵심 원칙 (답 먼저)

- BLUF / Pyramid Principle: 결론·권고·요청(ask)을 맨 위에. 분석은 bottom-up으로 하되 전달은 결론부터. 임원은 60초 안에 권고를 이해해야 한다. 권고가 12번째 장에 있으면 "추가 검토"로 죽는다.
- So-what 테스트: 모든 문장·슬라이드에 "그래서?"를 물어 함의가 남는지. 데이터 나열은 소음 - 함의·권고로 한 단계 올린다.
- MECE: 근거 그룹은 중복 0·누락 0, 3-4개.
- 한국 맥락: 두괄식, 한장보고(one-page) 기대, 쉬운 단어(전문용어는 각주), 정확한 서식 = 수신자에 대한 존중.
- 문체 일관: 한 문서 안에서 한 종결체로 통일한다 - 내부 보고는 개조식 평서체(~함/~임/~한다), 대외 제안은 합쇼체(~합니다). 의사결정 요청(ask)은 정중하게(~요청드립니다 / 승인 바랍니다). 섞으면 doc-critic이 문체 불일치로 막는다.

## Action title (제목 = 결론, 주제 아님)

슬라이드/섹션 제목을 주제 라벨이 아니라 완결된 결론 문장으로. 가능하면 수치 포함(모르면 플레이스홀더, 날조 금지):
- X "Q3 매출 개요" -> O "Q3 매출 [NN]% 성장, [지역] 확대가 견인"
- X "추진 배경" -> O "분산 관리가 재사용·품질·출제 효율을 저해"
- Horizontal logic: 제목만 위에서 아래로 읽어 스토리가 되는지 테스트(titles test). Vertical logic: 본문/차트가 그 제목을 실제로 증명(제목이 본문보다 더 약속하면 안 됨).
- 한 슬라이드 = 한 메시지. 제목이 두 결론을 담으면 폰트를 줄이지 말고 슬라이드를 쪼갠다.

## Storyline-first (ghost deck)

디자인 전에 흐름부터. 빈 화면에 action title들만 써서 위->아래로 읽어 논리를 검증한 뒤 본문·차트를 채운다. 답 먼저(Pyramid) -> action title 흐름 -> 슬라이드당 1메시지 -> 그다음 디자인.

## Executive summary (1페이지, decision-ready)

임원이 이것만 읽어도 결정 가능해야. SCQA로 도입 + 권고 먼저:
- S(상황): 모두 동의하는 현재 상태(논쟁 없음).
- C(복잡): 그 안정을 깬 변화·문제·위협.
- Q(질문): 자연히 떠오르는 "그래서 무엇을 해야?".
- A(답): 권고/결론 = 본론의 꼭대기.

순서: 목적/훅 -> 권고(ask 먼저) -> 핵심 발견(각 항목 = 하나의 숫자, 추세 서술 X) -> 비즈니스 임팩트 -> 실행 로드맵 -> CTA. 관찰(observation)과 의견(opinion)을 섞지 말 것.

## Status report 7섹션 (진행 현황 보고)

backward-looking 기록이 아니라 forward-looking 의사결정 도구. 순서:
1. 헤더 - 프로젝트/보고일/기간/담당 + **전체 RAG**
2. Executive summary - 2-3문장 TL;DR(시니어는 이것만 읽음)
3. RAG 대시보드 - 일정/예산/범위/자원/리스크 각각 색
4. 진척·핵심 성과 - 완료 산출물·마일스톤·KPI 수치
5. 리스크·이슈 - 각 리스크 + **완화책 + 담당**
6. 다음 단계 - 할 일 + 담당 + 마감
7. **필요 의사결정(Decisions Needed)** - 독립 섹션, 구체 ask. 이게 있어야 보고가 행동을 끌어낸다.

빼라: 활동(activity) 나열, 결론 없는 데이터, 장문 문단 -> 불릿·표로. client-facing은 1페이지.

## RAG status (신호등, 주관 제거)

정량 임계값으로: 편차 <5% Green / 5-15% Amber / >15% Red(또는 조직 기준). Amber는 머무르지 않게(통제로 복귀 또는 방치로 Red 전락). 색만 쓰지 말고 narrative 병기(상태+이슈+영향+조치). 색맹 대비 라벨 redundancy. 하우스 팔레트: Green `#2E6F5E` / Amber `#E3A82B`(밝은 골드 + 다크 텍스트 - 어둡게 누르면 갈색=AI slop) / Red `#B0413E` - 전부 contrast-gate AA.

## 차트 (메시지 운반체)

- 차트에도 결론 제목(차트 라벨 X). 차트 1개 = 메시지 1개.
- Zelazny 비교 유형 -> 차트: 시간변화 = line/column, 항목순위 = bar(수평), 구성비 = stacked/column(**pie 회피** - 24% vs 28% 구분 불가), 분포 = histogram, 상관 = scatter, 복합지표(절대값+비율) = 이중축 콤보(막대+선, 보조축), 증감 분해 = 워터폴(예산 대비 실적). 콤보·워터폴 구현은 `reference/xlsx.md`(openpyxl 보조축).
- Tufte data-ink: 격자선 옅게, 3D·그림자·테두리 제거. 핵심 막대 1개만 accent, 나머지 회색.
- 축·단위(%, 원) 명시, 범례는 데이터 안 가리게.

## 브랜드킷 적용 (회사 스타일)

메시지 구조는 회사 무관 동일, 색·폰트·로고·푸터만 brand-kit으로 덮어쓴다. RAG 임계값·action title 작법은 브랜드와 무관하니 바꾸지 말 것. 회사 색이 contrast-gate AA에 떨어지면 `brand-kit.md`의 어두운 변형 제안을 쓰고, RAG 의미색(Green/Amber/Red)은 가독성 깨지지 않는 한 유지. 적용 방법(템플릿 상속 vs brand-kit.json)은 `brand-kit.md`.

## 전송 이메일 (첨부 커버, 선택)

보고서·덱을 보낼 때의 커버 이메일도 결론 먼저. 제목 = 결론(또는 핵심 ask), 첫 줄 = 요청/결론 한 문장, 다음 1-2줄 = 맥락, 마지막 = 첨부 안내. 6줄 이내로 - 길면 본문은 첨부로 미룬다. producer는 문서를 보낼 때 vault에 `email.txt`로 함께 낸다(office-gate 텍스트 검사 대상).

```
제목: [결론/ask 한 줄 - 예: 2분기 파일럿 예산 승인 요청]

[수신 직책]께,

[핵심 결론·요청 한 문장]. 첨부 보고서의 핵심은 [한 줄]입니다.
[맥락 1-2줄]. [결정 기한/다음 단계].

첨부: [파일명] ([형식], [페이지/장수])
```

## 적용 + 검증

디자인은 `pptx.md`(팔레트/배지/표)·`docx.md`·`xlsx.md`(공통 팔레트). 메시지 구조는 위. 검증: 업무 문서 `office-gate.sh`(safety+korean+integrity+contrast). 수치는 `facts.json` 출처 또는 플레이스홀더(날조 금지) - integrity-gate가 막는다.
